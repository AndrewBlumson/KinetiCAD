Phase 5: Boolean Operations (Union, Subtract, Intersect)
Phase 4 added per-part modification features. Phase 5 introduces operations between parts: combine two or more parts into a single result through boolean union, subtraction, or intersection. The result becomes a new part (or replaces the inputs, configurable). After this phase, the modeller is complete.
Conceptual model:
Boolean operations are a special kind of feature that takes multiple input parts. To keep the data model consistent with the existing per-part feature tree, Phase 5 introduces a new top-level concept: a CompoundFeature at the assembly level rather than the part level.
Add to schemas:
tstype BooleanOperation =
  | { type: 'union' }
  | { type: 'subtract'; toolPartId: string }   // toolPartId is the part being subtracted from the others
  | { type: 'intersect' };

type BooleanFeature = {
  id: string;
  type: 'boolean';
  operation: BooleanOperation;
  inputPartIds: string[];        // 2 to 8 parts
  resultPartName: string;
  hideInputs: boolean;            // if true, input parts are hidden in the scene; if false, both inputs and result are visible
};

// Add to Assembly:
type Assembly = {
  // ... existing fields
  parts: Part[];
  booleanFeatures: BooleanFeature[];   // ordered, regenerated top to bottom
};
The persist version bumps from 3 to 4. Migration: existing assemblies have no booleanFeatures field; add as [].
Why a separate top-level structure rather than a feature on a part:
A boolean is a relationship between parts, not a property of one part. Phase 6 (multi-part scene management) and Phase 7 (mate joints) will operate on parts, and we want booleans to compose cleanly with those. Keeping booleans at the assembly level means input parts remain first-class entities in the parts panel and feature tree, which is essential for editing them.

Boolean operation implementation:
tsfunction booleanOp(shapes: TopoDS_Shape[], operation: BooleanOperation): TopoDS_Shape {
  if (shapes.length < 2) throw new Error('boolean requires at least 2 inputs');

  let result: TopoDS_Shape;

  if (operation.type === 'union') {
    // Fuse all shapes into one
    result = shapes[0];
    for (let i = 1; i < shapes.length; i++) {
      const fuser = new oc.BRepAlgoAPI_Fuse_3(result, shapes[i]);
      fuser.Build();
      if (!fuser.IsDone()) {
        // Retry with fuzzy tolerance
        fuser.delete();
        const fuzzyFuser = new oc.BRepAlgoAPI_Fuse_3(result, shapes[i]);
        fuzzyFuser.SetFuzzyValue(0.001);
        fuzzyFuser.Build();
        if (!fuzzyFuser.IsDone()) {
          fuzzyFuser.delete();
          throw new Error('boolean-failed');
        }
        const next = fuzzyFuser.Shape();
        fuzzyFuser.delete();
        if (i > 1) result.delete();
        result = next;
      } else {
        const next = fuser.Shape();
        fuser.delete();
        if (i > 1) result.delete();
        result = next;
      }
    }
  } else if (operation.type === 'subtract') {
    // shapes[0] is the tool, shapes[1..] are the bodies being cut from
    // Actually, conceptually: tool is the part doing the cutting
    // The convention here: inputPartIds[0] is the body, toolPartId is the cutter
    // shapes[] is built in inputPartIds order; toolPartId may be at any index
    // The caller must ensure the shapes array is ordered: [body, tool] for subtract
    const body = shapes[0];
    const tool = shapes[1];
    if (shapes.length > 2) throw new Error('subtract requires exactly 2 inputs');
    const cutter = new oc.BRepAlgoAPI_Cut_3(body, tool);
    cutter.Build();
    if (!cutter.IsDone()) {
      cutter.delete();
      const fuzzyCutter = new oc.BRepAlgoAPI_Cut_3(body, tool);
      fuzzyCutter.SetFuzzyValue(0.001);
      fuzzyCutter.Build();
      if (!fuzzyCutter.IsDone()) {
        fuzzyCutter.delete();
        throw new Error('boolean-failed');
      }
      result = fuzzyCutter.Shape();
      fuzzyCutter.delete();
    } else {
      result = cutter.Shape();
      cutter.delete();
    }
  } else if (operation.type === 'intersect') {
    // shapes[0] AND shapes[1] AND ... shapes[n]
    result = shapes[0];
    for (let i = 1; i < shapes.length; i++) {
      const intersector = new oc.BRepAlgoAPI_Common_3(result, shapes[i]);
      intersector.Build();
      if (!intersector.IsDone()) {
        intersector.delete();
        const fuzzyIntersector = new oc.BRepAlgoAPI_Common_3(result, shapes[i]);
        fuzzyIntersector.SetFuzzyValue(0.001);
        fuzzyIntersector.Build();
        if (!fuzzyIntersector.IsDone()) {
          fuzzyIntersector.delete();
          throw new Error('boolean-failed');
        }
        const next = fuzzyIntersector.Shape();
        fuzzyIntersector.delete();
        if (i > 1) result.delete();
        result = next;
      } else {
        const next = intersector.Shape();
        intersector.delete();
        if (i > 1) result.delete();
        result = next;
      }
    }
  } else {
    throw new Error('Unknown boolean operation');
  }

  return result;
}
Edge cases to handle:

Empty intersection (shapes don't overlap at all): OCCT may return a valid but empty shape. Detect via BRepCheck_Analyzer or by checking the result shape's bounding box volume; if zero, return 'empty-result' error.
Subtract producing empty result (tool fully contains body): same detection, surface as 'empty-result'.
Open shells from upstream features: error as 'invalid-input-not-solid'.

Note for Agent:
The tool part for Subtract is the part that "drills" into the others. UX-wise the inputs list has checkboxes (which parts participate) and the tool is a radio (which one is the cutter). The cutter must also be in the inputs list. This is two-step UX: first decide which parts participate, then if Subtract pick which is the tool. If Union or Intersect is selected, the tool radio is hidden.
The fuzzy-retry pattern (try with default tolerance, fall back to 0.001 fuzzy) is critical for booleans. Default tolerance fails on tight intersections more than on extrudes or fillets. Without fuzzy fallback, you'll see "boolean-failed" errors on geometry that should obviously work.
Do not auto-delete input parts when hideInputs is true. Only hide them. The user might toggle hideInputs off later, and the inputs need to still exist. Cascade deletion only happens when the user explicitly deletes an input part that's used by a boolean.
The 8-input cap on Union and Intersect is a sanity guard. OCCT booleans are O(n) per pair, and with 8 inputs you do 7 pairwise operations. Above that, performance degrades and failure rates increase. If a user genuinely needs 10 inputs, they can chain two Union features.
